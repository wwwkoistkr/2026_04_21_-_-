### 요구사항 ③ "모바일에서도 화면과 연동" ✅ 구현 가능

**현재 상태 (문제점)**
```html
<table width="680" style="max-width:680px;width:100%;">
  <!-- 고정 680px 너비 + 인라인 스타일만 -->
  <!-- @media query 없음 → 모바일에서 글씨 작게 축소됨 -->
</table>
```

**현재 `<meta viewport>` 는 O** (L350 에 설정되어 있음).
그러나 **`@media` 쿼리가 전혀 없음** → 모바일에서 680px 를 강제로 작게 축소.

**v2.8.0 반응형 추가안**
```css
<style>
  /* 모바일 (화면 ≤ 600px) */
  @media only screen and (max-width: 600px) {
    .container { width: 100% !important; max-width: 100% !important; }
    .news-card { padding: 14px 16px !important; }
    h3 { font-size: 20px !important; }       /* 24px → 20px */
    .summary-box { font-size: 18px !important; }  /* 22px → 18px */
    .header-title { font-size: 26px !important; } /* 32px → 26px */
  }
  /* Retina / 고해상도 (DPR ≥ 2) */
  @media only screen and (-webkit-min-device-pixel-ratio: 2) {
    body { -webkit-font-smoothing: antialiased; }
  }
</style>
```

**필요한 수정**
- `email_sender.py` `HTML_TEMPLATE` (L345~) 의 `<head>` 에 `<style>` 블록 추가 (약 25줄)
- 각 요소에 `class="news-card"` / `class="summary-box"` 등 **class 속성 추가** (약 10개)

**주의사항 (이메일 CSS 의 함정)**
- ⚠️ Gmail 웹/앱은 `<style>` 을 지원하지만, **Gmail 구 버전** 및 **Outlook 2007~2013** 에서는 무시됨
- ✅ 대응: **인라인 스타일을 기본값**으로 두고, `<style>` 은 **override** 로만 사용
- ✅ 이렇게 하면 지원 환경에서는 반응형, 미지원 환경에서는 기본 렌더링으로 fallback

**호환성 매트릭스**

| 클라이언트 | 반응형 지원 | Fallback |
|------------|-------------|----------|
| Gmail 웹 | ✅ O | - |
| Gmail 앱 (iOS/Android) | ✅ O | - |
| Apple Mail (iOS/macOS) | ✅ O | - |
| Outlook 365 웹 | ✅ O | - |
| Outlook 2016+ | ⚠️ 부분 | 인라인 스타일로 fallback |

### 요구사항 ④ "해상도는 고해상도로 8K 정도" ⚠️ 재해석 필요

**기술적 팩트 체크**
> 이메일 본문은 **픽셀 그래픽이 아닙니다**. 순수 HTML 텍스트 + CSS 이므로 "해상도" 개념이 **존재하지 않습니다**.

**"8K" 의 3가지 가능한 해석**

#### 해석 A: "더 크고 선명한 글자/이미지" ✅ 가능
- 사용자님의 실제 의도일 가능성 높음
- **Retina/고해상도 디스플레이 최적화**로 달성 가능
- 구현:
  1. `-webkit-font-smoothing: antialiased` — 안티앨리어싱 (iPhone/Mac 지원)
  2. `text-rendering: optimizeLegibility` — 커닝/리가처 최적화
  3. 카테고리 뱃지·랭크 뱃지 배경을 CSS 그라디언트(벡터)로 렌더 → **무한 해상도**
  4. 이모지는 SVG 기반(OS 제공) → **자동으로 4K/8K 모니터에서 선명**

#### 해석 B: "해상도 높은 이미지 첨부" ⚠️ 제한적
- 이메일 클라이언트는 **최대 첨부 크기 20MB** (Gmail)
- 8K 이미지 1장 = **약 30~50MB** → 전송 실패 위험
- **권장하지 않음**

#### 해석 C: "웹 UI(Cloudflare Pages) 도 8K 대응" ✅ 가능
- 관리 UI(https://morning-stock-briefing.pages.dev)의 경우
- 현재는 일반 반응형 → **8K 모니터(7680×4320)** 에서도 자동 스케일링
- 추가 작업 불필요 (이미 지원됨)

**결론 (권장)**: **해석 A** 방식으로 "선명도 개선"을 적용. 실제 "8K 픽셀 이미지"는 이메일 용량 제약으로 부적합.

## ⚠️ 3. 리스크 및 완화 방안

### 리스크 ① 이메일 용량 증가
- **현재**: 약 50KB/통 (v2.7.0, 10건 뉴스 + 4줄 포맷)
- **v2.8.0 예상**: 약 **95~120KB/통** (7줄 + 확대 스타일)
- **Gmail 권고**: 102KB 초과 시 **본문 일부 잘림**("Message clipped")
- **완화**:
  - ✅ 인라인 스타일 → CSS 클래스 변환으로 **반복 스타일 제거** (약 20KB 절감)
  - ✅ 이미지 없이 순수 HTML 유지
  - ✅ 예상 최종 용량 **60~80KB** (Gmail 한계 이하)

### 리스크 ② OpenAI 토큰 비용 증가
- **현재 (v2.7.0)**: 뉴스 1건 약 1,000 input + 500 output 토큰 = 월 $0.15
- **v2.8.0 예상**: 약 1,200 input + 900 output 토큰 = 월 **$0.24** (+$0.09 ≈ 130원)
- **완화**: 추가 비용 미미, OpenAI $5 초기 크레딧으로 **20개월 커버 가능**

### 리스크 ③ 7줄 중 일부가 비어있을 가능성
- AI가 숫자가 부족한 뉴스에는 📉 또는 🎯 라인을 채우지 못할 수 있음
- **완화**: validator 가 `"정보 부족"` 문구를 허용하되 **최소 5줄 이상 검증** 로 완화

### 리스크 ④ 모바일 CSS 호환성
- 일부 구 버전 Outlook 에서 `@media` 무시
- **완화**: 인라인 스타일을 기본값으로, `@media` 는 override 역할만 수행
