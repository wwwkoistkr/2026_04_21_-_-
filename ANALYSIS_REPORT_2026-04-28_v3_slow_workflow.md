## 2. 실행 시간 분석

### 예상 실행 시간 계산 (단일 파이프라인 `python main.py all`)

| 단계 | 내용 | 예상 소요 시간 |
|---|---|---|
| **환경 셋업** | Python 3.11 설치 + pip install | ~30~60초 |
| **Stage 1: Collect** | KV 소스 수집 + 미국 뉴스 RSS + 기사 스크래핑 | ~30~90초 |
| **Stage 2: Summarize** | Gemini AI 랭킹(1회) + 개별 요약(15회) | ⭐ **~90~180초** |
| **Stage 3: Send** | HTML 생성 + Gmail SMTP 발송 | ~10~30초 |
| **합계** | | **약 3~6분** |

### 🔴 핵심 병목: AI 요약 단계 (Stage 2)

```python
TARGET_NEWS_COUNT = 15              # 요약할 뉴스 개수
SUMMARY_MODE = "sequential"         # 순차 모드 (기본)
SUMMARY_CALL_DELAY_SEC = 6          # 각 호출 사이 6초 대기
```

**순차 모드 최소 대기 시간 계산:**
```
= (15건 - 1) × 6초(딜레이) + 15건 × ~3~5초(API 응답)
= 14 × 6 + 15 × 4
= 84 + 60
= ~144초 (약 2분 24초) — AI 요약만
```

**총 AI 호출 시간 (랭킹 포함):**
```
= 1회(랭킹) + 15회(개별 요약) + 1회(총평) + 1회(자가평가)
= 18회 × (6초 딜레이 + ~4초 응답)
= 약 180초 (3분) — AI 호출만으로
```

**실제 전체 워크플로우:**
```
GitHub 큐 대기:      ~10~30초
환경 셋업:           ~30~60초
수집 (Stage 1):      ~30~90초
AI 요약 (Stage 2):   ~150~240초 ⭐ 병목
발송 (Stage 3):      ~10~30초
────────────────────────────────
합계:                ~4~8분
```

## 4. 최적화 방안

### 🥇 방안 1: AI 호출 간격 단축 (가장 효과적, 즉시 적용 가능)

**현재**: `SUMMARY_CALL_DELAY_SEC = 6` (10 RPM 안전 마진)

**Gemini 무료 티어 실제 제한**: 15 RPM (= 4초 간격)  
**Gemini 유료 티어**: 1000+ RPM (딜레이 불필요)

| 변경안 | 딜레이 | AI 단계 예상 시간 | 총 절감 |
|---|---|---|---|
| 현재 | 6초 | ~180초 | — |
| **안 A** | **4초** | ~120초 | **~60초 절감** |
| **안 B** | **3초** | ~90초 | **~90초 절감** |
| 안 C (병렬) | 0초 | ~30초 | ~150초 절감 (쿼터 주의) |

**적용 방법** (GitHub Secrets):
```
SUMMARY_CALL_DELAY_SEC = 4    # 6초 → 4초 (안전한 선택)
```

### 🥈 방안 2: 병렬 모드 전환

```
SUMMARY_MODE = "parallel"          # sequential → parallel
SUMMARY_MAX_WORKERS = 4            # 동시 4개 실행
```

**장점**: AI 요약 시간 180초 → ~45초 (75% 절감)  
**위험**: Gemini 15 RPM 초과 시 429 에러 → **유료 티어에서만 권장**

### 🥉 방안 3: 뉴스 개수 축소

```python
TARGET_NEWS_COUNT = 15  →  12 또는 10
```

| 개수 | AI 호출 | 순차 6초 딜레이 예상 |
|---|---|---|
| 15건 | ~18회 | ~180초 |
| **12건** | ~15회 | **~150초** |
| **10건** | ~13회 | **~130초** |

### 방안 4: 기사 본문 스크래핑 비활성화 (수집 단계 최적화)

```
ARTICLE_SCRAPE_ENABLED = false
```
→ 수집 단계에서 약 30~60초 절감. 단, 요약 품질이 약간 저하될 수 있음.

### 방안 5: pip 캐시 최적화 (환경 셋업 최적화)

현재 `setup-python@v5`에 `cache: "pip"`이 설정되어 있어 이미 캐시됨.
추가로 `requirements.txt` 해시 기반 캐시가 작동 중 → 추가 최적화 여지 적음.

## 5. 권장 조치 (코드 수정 없이 즉시 적용)

### 즉시 적용: GitHub Secrets 변경

```bash
# GitHub 리포 → Settings → Secrets → Actions에서:
SUMMARY_CALL_DELAY_SEC = 4    # 6초 → 4초
```

→ 이 한 줄만으로 **전체 실행 시간 약 1분 단축** (8분 → 7분 이하)

### 코드 수정이 필요한 최적화

| 방안 | 예상 절감 | 난이도 | 위험도 |
|---|---|---|---|
| `DELAY = 4초` (Secrets) | ~1분 | ⭐ 즉시 | 🟢 낮음 |
| `DELAY = 3초` (Secrets) | ~1.5분 | ⭐ 즉시 | 🟡 중간 (RPM 근접) |
| 병렬 모드 전환 | ~2.5분 | Secrets 변경 | 🔴 쿼터 초과 위험 |
| `TARGET_NEWS_COUNT = 12` | ~30초 | 코드 수정 | 🟢 낮음 |
| 스크래핑 비활성화 | ~30~60초 | Secrets 변경 | 🟡 품질 저하 |
